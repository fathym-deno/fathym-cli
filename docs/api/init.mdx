# ftm init

Scaffold a new CLI project from templates.

## Usage

```bash
ftm init [name] [flags]
```

## Arguments

| Argument | Required | Default | Description |
|----------|----------|---------|-------------|
| `name` | No | `.` | Project name or path. Use `.` for current directory. |

## Flags

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--template` | string | `init` | Template to use for scaffolding |
| `--baseTemplatesDir` | string | (injected) | Root directory for templates |
| `--targetDir` | string | (cwd) | Where to scaffold the project |

## Examples

```bash
# Initialize new CLI project
ftm init my-cli

# Initialize in current directory
ftm init .

# Use custom template
ftm init my-cli --template=minimal

# Scaffold to specific directory
ftm init my-cli --targetDir=./projects
```

## Generated Structure

When using the default `init` template:

```
my-cli/
├── .cli.json           # CLI identity (Name, Tokens, Version)
├── .cli.init.ts        # IoC initialization hook
├── deno.jsonc          # Deno configuration and tasks
├── .gitignore          # Standard ignores (.build/, .dist/, etc.)
├── .vscode/
│   └── settings.json   # IDE configuration for Deno
├── commands/
│   ├── hello.ts        # Sample fluent API command
│   └── wave.ts         # Sample class-based command
└── intents/
    ├── .intents.ts     # Intent test index
    ├── hello.intents.ts
    └── wave.intents.ts
```

## How It Works

1. Parse arguments and flags via Zod schemas
2. Resolve target DFS (execution directory or `--targetDir`)
3. Initialize `TemplateScaffolder` with template locator
4. Scaffold project from selected template (default: `init`)
5. Log success with output path
